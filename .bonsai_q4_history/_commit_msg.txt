Transposed V cache layout from [head][time][dim] to [head][dim][time] across all 7 kernels that access V cache (3 GQA attention variants + 4 KV cache update variants). K cache layout unchanged.

**Problem**: In GQA Phase 5 (weighted V sum), SIMD lanes iterate over time steps t, reading vc[t*head_dim+d]. Adjacent lanes access addresses head_dim×2=256 bytes apart — each lane hits a separate 128-byte cache line. 32 lanes load 32 cache lines (4096 bytes) for 64 bytes of useful data = 1.6% bandwidth utilization.

**Fix**: With transposed layout vc[d*max_ctx+t], adjacent lanes read consecutive addresses (2 bytes apart). 32 values fit in one cache line. 128 bytes loaded for 64 bytes useful = 50% utilization (32× improvement).

**Estimated savings**: ~63KB wasted bandwidth per head per block × 16 heads × 28 blocks = ~28MB per token. V cache writes become strided but happen once per KV head vs reads happening once per query head (4× fewer), and write volume is tiny (128 elements per head).

**Correctness**: Golden output test passes — all 7 V cache access points updated consistently (5 write sites, 3 read sites).
